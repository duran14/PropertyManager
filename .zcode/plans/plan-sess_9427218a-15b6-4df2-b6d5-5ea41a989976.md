# Plan: Módulo A — Journey completo del inquilino (prospección → firma)

## Visión
Expandir el Módulo A de "capturar leads" a gestionar **todo el journey**: fotos IA → publicación → chatbot omnicanal → agendamiento ShowMojo → firma DocuSign. Cada fase entrega valor independiente y sigue el flujo del inquilino.

## Decisiones de integración (investigadas y confirmadas)
- **Fotos**: Autoenhance.ai (enhance + object removal + virtual staging, una sola API con webhooks)
- **Chatbot omnicanal**: Twilio (WhatsApp+SMS) + Telegram Bot API + widget web embebido + email reactivo
- **Agendamiento**: ShowMojo (push de listings vía `POST /api/v1/listings`, recepción de showings vía webhook)
- **Firma**: DocuSign eSignature REST API (legalmente válido en BC per Electronic Transactions Act)

---

## Fase 8 — Fotos IA (Autoenhance.ai)
*El PM sube fotos → la IA las mejora → se asocian a la unidad para publicación.*

**Backend:**
- Nuevo contrato `PhotoEnhancementAdapter`: `enhance(imageUrl, type)`, `removeObjects(imageUrl)`, `virtualStage(imageUrl, style)`, `parseWebhook(headers, body)`
- Mock que devuelve URLs de imágenes de demo
- Nuevo modelo `ListingPhoto` (unitId, originalUrl, enhancedUrl, type, status, autoenhanceOrderId)
- Job BullMQ `photo-enhancement` — sube imagen a Autoenhance, recibe webhook cuando termina
- Webhook `POST /webhooks/autoenhance` — recibe imagen procesada
- Rutas `POST /units/:id/photos` (subir), `POST /photos/:id/enhance` (disparar IA), `GET /units/:id/photos`

**Frontend:**
- Página "Galería de fotos" por unidad: subir, ver before/after, elegir tipo de mejora

---

## Fase 9 — Chatbot omnicanal con FSM real
*El bot conversa por WhatsApp/Telegram/web, califica con slots, propone unidades.*

**Cambios estructurales (afectan schema):**
- `LeadSource`: añadir `telegram`, `web`
- `ChatConversation`: `channel` → enum `ChatChannel { whatsapp, sms, telegram, web, email }`
- `Lead.conversationId`: quitar `@unique` (un lead puede tener varias conversaciones multi-canal)
- `ChatMessage`: añadir `mediaUrls String[]` (soporte de imágenes)
- Nuevo modelo `ConversationSlot` (conversationId, key, value — para budget, move-in date, etc.)
- De-duplicación de leads por phone/email (no por conversationId)

**Backend:**
- Nuevo contrato unificado `MessagingAdapter` con implementaciones: `TwilioAdapter` (WhatsApp+SMS), `TelegramAdapter`, `WebChatAdapter`
- **FSM conversacional real** con GLM estructurado (responseSchema): GLM devuelve `{intent, reply, slots, next_state}` en vez de texto libre
- Estados: `greeting → collecting_budget → collecting_movein → proposing_units → proposing_tour → scheduling → handoff`
- El chatbot ahora usa el historial completo (no solo el último mensaje)
- Ruta `POST /chat/messages` para web chat embebido (con auth de sesión anónima)
- Webhook `POST /webhooks/telegram`
- Email reactivo: si el lead contactó por email, el bot responde por email

**Frontend:**
- Widget de chat embebido en la página pública de unidad (`/public/units/:slug`)
- Panel de conversación en la página de Leads (ver historial completo, intervenir/manual override)

---

## Fase 10 — Agendamiento con ShowMojo
*El bot deriva a ShowMojo para agendar; recibimos confirmaciones vía webhook.*

**Backend:**
- Nuevo contrato `ShowMojoAdapter`: `pushListing(unitData)`, `getAvailableSlots(listingCode)`, `createShowing(listingCode, slot, prospect)`, `parseWebhook(headers, body)`
- Mock que devuelve slots y showings simulados
- Nuevo modelo `Showing` (leadId, unitId, showmojoId, scheduledAt, brokerUserId, status)
- `Lead`: añadir `showmojoShowingId`, `tourUrl`
- Cuando el FSM llega a `scheduling`: llama ShowMojo para obtener slots → propone al lead → confirma
- Webhook `POST /webhooks/showmojo` ampliado: `lead_captured`, `showing_scheduled`, `showing_confirmed`, `showing_cancelled`
- Notificación al broker/PM (vía el canal del chatbot o email) cuando se agenda una visita
- Ruta `GET /leads/:id/showing`, `POST /leads/:id/schedule`

**Frontend:**
- En la página de Leads: mostrar visita agendada, enlace a ShowMojo, estado de confirmación del broker
- Botón "Confirmar visita" para el broker/PM

---

## Fase 11 — Firma de contratos con DocuSign
*Cuando el lead convierte → genera RTA → envía a DocuSign → recibe firmado.*

**Backend:**
- Nuevo contrato `DocuSignAdapter`: `createEnvelopeFromRtaDraft(leaseId)`, `getEnvelopeStatus(envelopeId)`, `parseWebhook(headers, body)`
- Mock que simula el flujo de firma
- `Lease`: añadir `docusignEnvelopeId`, `docusignStatus`
- Flujo: `generateRtaDraft` → `createEnvelopeFromRtaDraft` → DocuSign envía email al inquilino → webhook `envelope-completed` → `Lease.signedDocRef` se popula, `status: active`
- Webhook `POST /webhooks/docusign` para `envelope-sent`, `envelope-completed`, `envelope-declined`
- Ruta `POST /leases/:id/send-for-signature`, `GET /leases/:id/signature-status`

**Frontend:**
- En página de Contratos/RTA: botón "Enviar a DocuSign" (además de "Generar RTA")
- Indicador de estado de firma (enviado/visualizado/firmado/rechazado)
- Cuando el broker confirma, avanza a "Enviar a firma" automáticamente

---

## Fase 12 — Pipeline integrado + publishing
*Conecta todo: fotos mejoradas → push a ShowMojo → listing publicado.*

**Backend:**
- Servicio `publishUnit(unitId, channels)`: toma fotos mejoradas + datos de la unidad → push a ShowMojo
- Cuando una unidad se publica, su URL pública queda activa con la galería de fotos mejoradas
- Dashboard de prospección: métricas del funnel (leads → tours → conversiones)

**Frontend:**
- Página "Publicar unidad": elegir fotos, seleccionar canales, confirmar publicación
- Mejora del Dashboard con funnel de conversión visual

---

## Orden de ejecución recomendado
F8 (Fotos) → F9 (Chatbot omnicanal) → F10 (ShowMojo) → F11 (DocuSign) → F12 (Pipeline)

Cada fase es independiente y deja el sistema funcional. Las fases 8 y 9 pueden hacerse en paralelo si se prefiere.

## Migraciones de schema necesarias
- `LeadSource`: +`telegram`, +`web`
- `ChatConversation.channel`: String → enum ChatChannel
- `Lead.conversationId`: quitar @unique
- `ChatMessage`: +`mediaUrls String[]`
- Nuevos modelos: `ListingPhoto`, `ConversationSlot`, `Showing`
- `Lead`: +`showmojoShowingId`, +`tourUrl`
- `Lease`: +`docusignEnvelopeId`, +`docusignStatus`

## Lo que NO incluye (post-MVP)
- Firma de todos los formularios provinciales (solo RTA-1 en MVP)
- Sincronización bidireccional de calendario (ShowMojo gestiona su propio calendario)
- Webhooks con firma verificada (se añade en endurecimiento de seguridad, fase separada)
- Portales de inquilinos/propietarios

## Estimación
Es un expansion significativa (~5 fases). Cada fase es 1-2 sesiones de trabajo. Recomiendo empezar por F8+F9 en paralelo.
